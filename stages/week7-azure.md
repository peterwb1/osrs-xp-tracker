# Weekend 7 — Deploy to Azure

## Goal

By Sunday the app is live on the internet. Someone at any computer can open a URL, register, and start tracking their OSRS character — served from Azure infrastructure, not your laptop. The backend runs in Azure Container Apps, the database is Azure Database for PostgreSQL, and the React frontend is on Azure Static Web Apps. You'll have a real domain (auto-generated by Azure) and HTTPS for free.

## Starting Point

- `docker-compose.yml` works locally (`docker compose up --build`)
- Both Dockerfiles are optimised (Weekend 6 done)
- An Azure account exists (free tier or pay-as-you-go)
- Azure CLI installed: `az --version`

## Choices

### 1. Container registry: Azure Container Registry (ACR) vs GitHub Container Registry (GHCR)

**Option A — Azure Container Registry (ACR)**
```bash
# Create registry (Basic tier ~£4/month)
az acr create --resource-group osrs-tracker-rg --name osrstrackerpeter --sku Basic --location uksouth

# Build and push
az acr login --name osrstrackerpeter
docker build -t osrstrackerpeter.azurecr.io/api:latest ./api
docker push osrstrackerpeter.azurecr.io/api:latest

# Grant Container Apps access to ACR
az acr update --name osrstrackerpeter --admin-enabled true
```
Pros: Everything in one Azure resource group, integrated with Container Apps identity.
Cons: Costs ~£4/month even for the Basic tier, even with no traffic.

**Option B — GitHub Container Registry (GHCR) — recommended**
```bash
# Login with your GitHub PAT (needs write:packages scope)
echo $GITHUB_TOKEN | docker login ghcr.io -u peterwb1 --password-stdin

# Tag and push
docker build -t ghcr.io/peterwb1/osrs-tracker-api:latest ./api
docker push ghcr.io/peterwb1/osrs-tracker-api:latest
```
Then in Container Apps, reference `ghcr.io/peterwb1/osrs-tracker-api:latest` and provide a registry credential (username = your GitHub username, password = a PAT with `read:packages`).

Pros: Free for public repos, very cheap for private (~£0.001/GB storage), integrated with GitHub Actions in Weekend 8.
Cons: Extra step to configure Container Apps with GHCR credentials.

**Recommendation:** GHCR if your repo is public. ACR if you want everything in Azure and don't mind the cost.

### 2. Database: Azure Database for PostgreSQL Flexible Server tiers

**Option A — Burstable B1ms (cheapest — recommended for this project)**
- 1 vCore, 2GB RAM, 32GB storage
- ~£10/month (12 months free on new Azure accounts)
- Good enough for dozens of users and millions of XP snapshots

```bash
az postgres flexible-server create \
  --resource-group osrs-tracker-rg \
  --name osrs-tracker-db \
  --admin-user osrs \
  --admin-password "YourStrong!Password123" \
  --sku-name Standard_B1ms \
  --tier Burstable \
  --version 16 \
  --location uksouth \
  --public-access None
```
Note: `--public-access None` means only Azure services can connect. You'll need to add a firewall rule for your Container Apps.

**Option B — Allow public access (simpler for dev)**
```bash
az postgres flexible-server create \
  --resource-group osrs-tracker-rg \
  --name osrs-tracker-db \
  --admin-user osrs \
  --admin-password "YourStrong!Password123" \
  --sku-name Standard_B1ms \
  --tier Burstable \
  --version 16 \
  --location uksouth \
  --public-access 0.0.0.0
```
`0.0.0.0` allows all Azure services (not the open internet, but not locked to your Container App).
Pros: No need to figure out Container Apps outbound IPs.
Cons: Less secure.

**Recommendation:** Option B for Weekend 7 — simplifies setup while keeping it Azure-internal. Restrict properly in production.

### 3. Container Apps: deployment method

**Option A — Azure CLI (one-time, manual)**
```bash
# Create Container Apps environment
az containerapp env create \
  --name osrs-tracker-env \
  --resource-group osrs-tracker-rg \
  --location uksouth

# Deploy API container
az containerapp create \
  --name osrs-tracker-api \
  --resource-group osrs-tracker-rg \
  --environment osrs-tracker-env \
  --image ghcr.io/peterwb1/osrs-tracker-api:latest \
  --registry-server ghcr.io \
  --registry-username peterwb1 \
  --registry-password $GITHUB_TOKEN \
  --target-port 8080 \
  --ingress external \
  --min-replicas 1 \
  --max-replicas 3 \
  --secrets "db-conn=Host=osrs-tracker-db.postgres.database.azure.com;Port=5432;Database=osrstracker;Username=osrs;Password=YourStrong!Password123;Ssl Mode=Require" \
            "jwt-key=your-production-jwt-secret-key-here" \
  --env-vars "ConnectionStrings__Default=secretref:db-conn" \
             "Jwt__Key=secretref:jwt-key" \
             "Jwt__Issuer=OsrsTracker"
```

**Option B — Bicep / ARM templates (infrastructure-as-code)**
Define all Azure resources in a `infrastructure/main.bicep` file and deploy with `az deployment group create`. 
Pros: Repeatable, version-controlled, can be run in CI/CD.
Cons: Bicep has a learning curve, overkill for a single-person project at this stage.

**Option C — Azure Portal (GUI)**
Click through the Azure Portal to create each resource.
Pros: No CLI learning, visual feedback.
Cons: Not reproducible, easy to miss settings, can't automate in Weekend 8.

**Recommendation:** Azure CLI (Option A). It's repeatable (you can re-run the commands), not as complex as Bicep, and teaches you the Azure resource model. You'll need CLI knowledge for Weekend 8's CI/CD anyway.

### 4. Frontend: Azure Static Web Apps vs Azure Blob Storage + CDN

**Option A — Azure Static Web Apps (recommended)**
```bash
az staticwebapp create \
  --name osrs-tracker-web \
  --resource-group osrs-tracker-rg \
  --location westeurope \
  --source https://github.com/peterwb1/osrs-xp-tracker \
  --branch main \
  --app-location "/web" \
  --output-location "dist" \
  --login-with-github
```
This creates a GitHub Actions workflow that auto-builds and deploys `web/` on every push to `main`.
Free tier: 100GB bandwidth/month, custom domains, HTTPS, global CDN.

Configure SPA routing in `web/staticwebapp.config.json`:
```json
{
  "navigationFallback": {
    "rewrite": "/index.html",
    "exclude": ["/api/*", "*.{js,css,png,svg,ico}"]
  }
}
```
This replaces nginx's `try_files` — Static Web Apps handles SPA routing automatically.

**Option B — Azure Blob Storage + Azure CDN**
Upload `dist/` to a Blob Storage container with "Static website" enabled, then add Azure CDN for caching and custom domain.
Pros: Cheaper at high scale (~£0.001/GB storage, £0.05/GB CDN transfer).
Cons: No automatic deployment, manual configuration for SPA routing, no built-in GitHub integration.

**Recommendation:** Azure Static Web Apps. Free tier is more than enough, GitHub integration is automatic, and SPA routing is built-in.

### 5. CORS: allowing the frontend origin

When the React app (on `https://calm-sand-001.azurestaticapps.net`) calls the API (on `https://osrs-tracker-api.niceregion.azurecontainerapps.io`), the browser blocks the request unless the API explicitly allows the origin.

**Option A — Specific origin in code (recommended for production)**
```csharp
// In Program.cs
builder.Services.AddCors(options => {
    options.AddPolicy("Frontend", policy => {
        policy.WithOrigins(
            "http://localhost:5173",        // Vite dev server
            "http://localhost:3000",         // Docker Compose
            builder.Configuration["Frontend:Url"] ?? "" // production URL from env var
        )
        .AllowAnyHeader()
        .AllowAnyMethod();
    });
});

app.UseCors("Frontend");
```
Set `Frontend__Url` as a Container Apps environment variable (not a secret).

**Option B — Wildcard (quick but insecure)**
```csharp
policy.AllowAnyOrigin().AllowAnyHeader().AllowAnyMethod();
```
Pros: Works immediately, no URL coordination needed.
Cons: Any website on the internet can make requests to your API. Not appropriate for an authenticated API.

**Recommendation:** Specific origin (Option A). Get your Static Web Apps URL first, then set it as the `Frontend__Url` environment variable.

## Research Topics

**What is Azure Container Apps?**
A managed platform for running containers without managing Kubernetes directly. You define a Container App with: which container image to run, CPU/memory limits, how many replicas, and ingress settings (port, internal/external). Azure handles load balancing, HTTPS termination (free Let's Encrypt cert), and scaling. Container Apps environments group multiple apps that can communicate on a private network.

**What is a Container Apps Secret?**
An encrypted environment variable. You create it with `--secrets "name=value"` and reference it in env vars with `secretref:name`. Secrets aren't shown in `az containerapp show` output and can be rotated without redeploying. Use them for database passwords, JWT keys, and any sensitive value.

**What is `Ssl Mode=Require` in the PostgreSQL connection string?**
Azure Database for PostgreSQL requires encrypted connections by default. Without `Ssl Mode=Require` (or `Ssl Mode=VerifyFull` for stricter verification), Npgsql will fail to connect. Azure provides a CA certificate you can download for full certificate verification, but `Require` is sufficient for most cases.

**What is Azure Static Web Apps `staticwebapp.config.json`?**
A configuration file that lives in your app's root (or `public/` folder) and controls Azure SWA's routing rules. The `navigationFallback` section tells SWA to serve `index.html` for all paths not matching static files — the equivalent of nginx's `try_files`. Without it, refreshing on `/accounts/1` returns 404.

**What is a Container Apps revision?**
Every time you update a Container App (new image, changed env vars), Azure creates a new immutable revision. By default, traffic switches to the new revision immediately (rolling update). You can keep the old revision running and split traffic between them for blue-green deployments. `az containerapp revision list` shows all revisions.

**Why does the API need to run EF migrations in production?**
Your `Program.cs` calls `db.Database.Migrate()` on startup. This runs any pending migrations against the Azure PostgreSQL database the first time the container starts, creating tables. Subsequent startups see no pending migrations and continue immediately. This is acceptable for a personal project; a larger team would run migrations as a separate deployment step to avoid startup races.
